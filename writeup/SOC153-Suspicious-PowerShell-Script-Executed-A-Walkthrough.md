# SOC153 — Suspicious PowerShell Script Executed: A Walkthrough

In this post, I’ll walk through my investigation of SOC153 — Suspicious PowerShell Script Executed, an alert generated by LetsDefend. The goal isn’t just to explain what happened, but to break down the analytic thought process: how I verified the alert, identified indicators of compromise (IOCs), and confirmed malicious activity.

## Alert Overview

On March 14, 2024, at 5:23 PM, LetsDefend triggered an alert for suspicious PowerShell execution on host Tony (IP: 172.16.17.206).

The alert details included:

- File Path: `C:\Users\LetsDefend\Downloads\payload_1.ps1`
- File Hash (SHA256): `db8be06ba6d2d3595dd0c86654a48cfc4c0c5408fdd3f4e1eaf342ac7a2479d0`

This initial information pointed to possible misuse of PowerShell, a common vector for script-based attacks.

## Initial Investigation

My first step was to verify the file hash using VirusTotal. The hash came back with multiple detections marking the file as malicious, categorized as a PowerShell script. This immediately aligned with the alert type.

Next, I checked Log Management for the targeted IP (`172.16.17.206`) and filtered events from March 14, 2024, around the alert timestamp of 5:23 PM. I often review events preceding the alert to look for indicators that led up to the execution.

At 5:22 PM, just one minute before the alert, the endpoint accessed:

`hxxps://files-ld[.]s3.us-east-2.amazonaws.com/payload_1[.]ps1`

This URL pointed to an external AWS server, a red flag indicating possible remote payload retrieval. Upon testing the URL in VirusTotal and Hybrid Analysis, both tools marked it as malicious.

## Deep Dive: External Communication

Further log investigation revealed a command execution allowing PowerShell to run a script from another external domain:

`hxxps://kionagranada[.]com/upload/sd2[.]ps1`

Successive logs confirmed outbound communication, specifically with the IP `161.22.46[.]148`. VirusTotal analysis identified this IP as malicious, typically associated with command-and-control (C2) behavior.

A GET request from the internal host to the C2 server was successful, confirming live external communication. The sequence strongly indicated that the host was compromised and actively executing a remote script payload.

## Applying the SOC Playbook

### Step 1: Identify Threat Type

Outbound connection patterns and PowerShell script execution confirmed unexpected outgoing internet traffic, a classic indicator of compromise.


### Step 2: Check Quarantine Status

Reviewing logs for `172.16.17.206`, no evidence suggested that the malware had been quarantined or remediated.


### Step 3: Analyze the Malware

VirusTotal and Hybrid Analysis results classified the file hash as a malicious PowerShell downloader, designed to execute remote content.


### Step 4: Check for C2 Activity

The successful GET request to `161.22.46[.]148` established confirmed access to a command-and-control server, signifying post-compromise communication.

### Step 5: Containment

Immediate isolation of host Tony (`172.16.17.206`) was initiated to halt further external communication and prevent lateral movement.

### Step 6: Artifacts

Key artifacts collected during the investigation included the file hash, suspicious URLs, IP addresses, and log entries related to PowerShell execution and outbound traffic.

## Lessons Learned & Conclusion

The timeline of events unfolded rapidly within minutes:

- 5:22 PM: Malicious script fetched from AWS.
- 5:23 PM: PowerShell executed payload leading to C2 communication.

This alert underscores how attackers leverage legitimate tools like PowerShell for stealthy, script-based execution. Visibility into outbound traffic and script execution monitoring is vital for early detection.
